## 📝 文档规范

### 1. 所有项目必须有 `docs` 文件夹

**原因**: 
- MAS（Multiple-AGENT-Studios）项目文档管理混乱，OCT 要规范
- 方便后续维护和协作
- 记录开发过程和决策

### 2. 文档分类

| 文件夹 | 用途 | 示例 |
|--------|------|------|
| `docs/任务列表/` | 每日开发任务 | `2026-03-09_任务列表.md` |
| `docs/技术规范/` | 开发规范、API 文档 | `项目规范.md`、`API 对接文档.md` |
| `docs/更新日志/` | 版本更新记录 | `更新日志.md` |

### 3. 文件命名规范

- **任务列表**: `YYYY-MM-DD_任务列表.md`
- **技术文档**: `文档名称.md`（中文，短横线分隔）
- **更新日志**: `更新日志.md`

## 🎨 代码规范

### 命名约定

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件夹 | 短横线 | `OpenClaw-Terminal` |
| 文件 | 短横线 | `ChatTab.tsx` |
| 组件 | 大驼峰 | `ChatTab`, `OptionBox` |
| 函数 | 小驼峰 | `handleSend`, `quickSend` |
| 常量 | 全大写 | `MAX_VISIBLE_MESSAGES` |
| 类型 | 大驼峰 | `ChatMessage`, `ChatTabProps` |

### package.json 规范

```json
{
  "name": "openclaw-terminal",      // 小写 + 短横线
  "version": "1.0.0",                // 语义化版本
  "description": "OpenClaw Terminal - Windows 超级系统助手"
}
```

## 🚀 开发流程

### 每日流程

1. **查看任务列表** - `docs/任务列表/`
2. **更新任务状态** - 完成的任务标记 ✅
3. **记录问题** - 遇到的问题记录到文档
4. **提交代码** - 每日至少提交一次

### 版本发布流程

1. 更新 `package.json` 版本号
2. 编写更新日志（`docs/更新日志.md`）
3. Git 提交 + Tag
4. 构建发布
